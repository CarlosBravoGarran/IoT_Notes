Chuleta interactiva de Docker Compose: estructura, patrones y orden de arranque

Anatomía de un servicio
services: mi_servicio: build: ./mi_servicio # construye desde Dockerfile image: ubuntu:22.04 # O usa imagen directamente ports: - "5000:5000" # host:contenedor environment: - HOST=0.0.0.0 - PORT=5000 networks: - mi-red volumes: - "./code:/etc/usr/src/app" # host:contenedor depends_on: mi_db: condition: service_healthy restart: unless-stopped networks: mi-red: driver: bridge # red nueva (solo en un compose) otra-red: external: true # red ya existente (otros composes)
Truco build vs image: usa build para tu propio código (con Dockerfile), image para imágenes externas como MySQL, Mosquitto, Nginx.
Healthcheck — cuándo está "listo" un servicio
healthcheck: test: ["CMD", "healthcheck.sh", "--connect"] start_period: 10s # tiempo de gracia al arrancar interval: 10s # cada cuánto se comprueba timeout: 5s # tiempo máximo por comprobación retries: 3 # fallos antes de marcar unhealthy
Ojo: ports expone al host. expose solo es visible entre contenedores de la misma red — no al exterior.
Las 3 condiciones de depends_on
service_started
El contenedor ha arrancado, pero puede que el proceso interno aún no esté listo.

Usar para: Mosquitto, servicios sin BD.
service_healthy
El contenedor pasa el healthcheck. Garantiza que el servicio está operativo.

Usar para: MySQL, PostgreSQL, cualquier BD.
service_completed_successfully
El contenedor terminó con código 0. Para tareas de inicialización o migraciones.

Usar para: scripts de setup, db-init.

# Ejemplo: microservicio que espera a que la BD esté healthy depends_on: dbservice: condition: service_healthy # Ejemplo: varios depends_on depends_on: mosquitto: condition: service_started dbservice: condition: service_healthy
Error clásico: usar service_started para MySQL. El contenedor arranca en segundos pero MySQL tarda 10-30s en estar listo. Sin service_healthy + healthcheck el microservicio intenta conectar antes de tiempo y falla.
Truco: si no controlas el Dockerfile del servicio (imagen externa), define el healthcheck en el compose con test: ["CMD", "mysqladmin", "ping", "-h", "localhost"].
Red nueva vs red externa
Crear red (driver: bridge)
Solo en un compose. Este compose es el "dueño" de la red.

networks: fic-services: driver: bridge
Unirse a red externa (external: true)
En el resto de composes. Se unen a la red ya creada.

networks: fic-services: external: true
Orden de arranque entre composes: el compose que crea la red debe arrancarse primero. Si arrancas un compose con external: true antes de que exista la red, falla.
Puertos — regla de los conflictos
Truco para múltiples BDs: todas las instancias MySQL escuchan en el puerto interno 3306, pero el puerto del host debe ser único en toda la máquina.
# sessions DB ports: - "3306:3306" # telemetry DB → puerto host distinto ports: - "3307:3306" # devices DB → puerto host distinto ports: - "3308:3306"
Comunicación entre servicios
Dentro de la misma red Docker, los contenedores se referencian por nombre de servicio, no por IP. Si el servicio se llama mosquitto, la dirección es mosquitto:1883. Por eso las variables de entorno usan nombres como MQTT_SERVER_ADDRESS=mosquitto.
Los 3 patrones del examen
Base de datos por servicio
Cada microservicio tiene su propia BD. Nadie accede a la BD de otro directamente — solo a través de su API.

Resultado: un compose por microservicio, cada uno con su dbservice.
Backend for Frontends
El frontend no habla directamente con los microservicios. Hay un backend intermedio (webapp_backend) que agrega y adapta las llamadas.

Resultado: compose del frontend con dos servicios: backend + frontend.
Paso de mensajes
Los dispositivos no llaman directamente a los microservicios. Se comunican a través de un broker de mensajes (Mosquitto/MQTT).

Resultado: Message Router como intermediario entre MQTT y HTTP.
División de ficheros compose en este proyecto
# compose 1 — crea la red (dueño de fic-services) docker-compose-message-router.yml └── mosquitto # broker MQTT └── message_router # intermediario MQTT↔HTTP └── network fic-services (driver: bridge) ← crea la red # compose 2 — se une a la red docker-compose-sessions-microservice.yml └── sessions_microservice └── sessions-dbservice └── network fic-services (external: true) # compose 3 — se une a la red docker-compose-telemetry-microservice.yml └── telemetry_microservice └── telemetry-dbservice └── network fic-services (external: true) # compose 4 — se une a la red docker-compose-devices-microservice.yml └── devices_microservice └── devices-dbservice └── network fic-services (external: true) # compose 5 — BFF, se une a la red docker-compose-frontend.yml └── webapp_backend # BFF └── webapp_frontend # Nginx/React └── network fic-services (external: true)
Orden de arranque entre composes: primero el message-router (crea la red), luego los microservicios, finalmente el frontend.
