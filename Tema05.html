01 — Qué son los contenedores y por qué existen

El problema que resuelven los contenedores

En entornos con muchas aplicaciones ejecutándose en los mismos servidores, cada app tiene dependencias (frameworks, librerías, versiones) que pueden entrar en conflicto. La solución primitiva fue usar máquinas virtuales (una app por VM), pero son pesadas — cada VM incluye un SO completo entero.

Máquinas Virtuales (VM)

  • Hipervisor gestiona múltiples VMs sobre el hardware
  • Cada VM incluye su propio SO completo (Linux, Windows Server…)
  • Arranque lento (minutos)
  • Alto consumo de RAM y disco
  • Aislamiento fuerte pero ineficiente
  • Una app por VM = recursos muy desaprovechados

Contenedores Docker

  • Comparten el kernel del SO anfitrión
  • Solo empaquetan la app + sus dependencias
  • Arranque muy rápido (segundos o menos)
  • Consumo mínimo de RAM y disco
  • Aislamiento mediante namespaces y cgroups de Linux
  • Muchos contenedores por host sin problemas de compatibilidad

Definición formal

Un contenedor es una unidad estándar de software que empaqueta el código y todas sus dependencias para que la aplicación se ejecute de forma rápida y confiable en cualquier entorno. Una imagen Docker es un paquete ligero, independiente y ejecutable que incluye: código, tiempo de ejecución, herramientas del sistema, librerías y configuraciones.

02 — Arquitectura interna de Docker

Los 3 componentes clave de Docker

⚠ Pregunta directa — componentes clave de Docker

Las respuestas correctas son: Imágenes, Contenedores y Docker Daemon.
Incorrectas: Hipervisor (es de VMs), Kernel dedicado (los contenedores comparten el kernel del host).

📦
Imágenes (Images)

Plantillas de solo lectura (inmutables) formadas por capas. A partir de ellas se crean los contenedores. Se almacenan en Docker Hub o registros privados.

🚀
Contenedores

Instancias en ejecución de una imagen. Tienen una capa de escritura encima de las capas de imagen (read-only). Múltiples contenedores pueden compartir la misma imagen.

⚙️
Docker Daemon

Proceso en segundo plano que gestiona el ciclo de vida de los contenedores. Expone una API REST. La CLI de Docker es su cliente principal.

Capas de la arquitectura Docker

CLI / API REST Interfaz de usuario (docker run, docker build…) y API REST para automatización
Docker Engine Funcionalidad adicional: redes, plugins, orquestación. Consume la API REST de containerd.
containerd + runc Runtime del contenedor. Gestiona el ciclo completo: pull de imagen → creación → arranque → parada → eliminación.
Linux OS (kernel) Namespaces (pid, net, mnt, uts, ipc) + cgroups para aislamiento y limitación de recursos. Base de todo.

Namespaces de Linux

Permiten aislar procesos que se ejecutan dentro del contenedor. Tipos: pid (procesos), net (red), mnt (sistema de archivos), uts (hostname), ipc (comunicación entre procesos).

Proyecto Moby

Paraguas de código abierto de Docker que separa la parte open source de la comercial. El motor Docker original era monolítico; Moby lo fragmentó en componentes reutilizables independientes (containerd, runc, libnetwork…).

03 — Imágenes, capas y Copy-on-Write

Cómo funcionan las imágenes Docker

Las imágenes son pilas de capas inmutables. Cada capa contiene solo los cambios respecto a la capa anterior. Docker usa un sistema de archivos de unión (Union FS) para presentarlos como un único sistema de archivos al contenedor.

Ejemplo: imagen Alpine + Nginx + App Web
CAPA 4 Container Layer (RW) Capa de escritura añadida al crear el contenedor. Todos los cambios en ejecución van aquí.
CAPA 3 App Web (HTML/CSS/JS) Archivos de la aplicación. Inmutable (read-only).
CAPA 2 Nginx Servidor web añadido sobre Alpine. Inmutable.
CAPA 1 (base) Alpine Linux Imagen base mínima. Inmutable. Normalmente viene de Docker Hub.
copy-on-write (CoW) — concepto clave de examen
  • Estrategia para compartir y copiar archivos con máxima eficiencia
  • Si una capa solo usa un archivo de una capa inferior → lo usa directamente
  • Si una capa quiere modificar un archivo de una capa inferior → lo copia primero a su capa y luego lo modifica
  • Las capas de imagen son siempre inmutables (read-only)
  • Solo la capa del contenedor (la más alta) es read/write
  • Múltiples contenedores pueden compartir las mismas capas de imagen
  • Resultado: uso mínimo de disco y arranque ultrarrápido
  • Controlador de gráficos (storage driver) gestiona la interacción entre capas

Docker Hub

Registro público de imágenes de contenedores. Repositorios privados y públicos. Builds automáticas desde GitHub/Bitbucket. Es el origen habitual de las imágenes base (Alpine, Ubuntu, CentOS, Node, Python…).

3 formas de crear una imagen

1. Interactiva — arrancar contenedor, modificar, hacer docker commit. Solo para prototipos.
2. Dockerfile — la forma correcta. Declarativa, repetible, escalable.
3. Importar tarball — importar desde un fichero .tar.

04 — Dockerfile

Instrucciones del Dockerfile

Un Dockerfile es un fichero de texto declarativo que describe cómo construir una imagen personalizada. Cada instrucción genera una nueva capa en la imagen resultante. Por convención las palabras clave van en MAYÚSCULAS.

FROM Define la imagen base desde la que se parte. Siempre la primera línea. FROM scratch = imagen vacía (sin generar capa). Ej: FROM node:9.4, FROM alpine.
RUN Ejecuta un comando dentro del contenedor durante la construcción. Ej: instalar paquetes (apt-get install, yum install, npm install). Cada RUN es una capa.
COPY Copia ficheros/carpetas del host a la imagen. Sintaxis: COPY <origen> <destino>. Solo archivos locales.
ADD Como COPY pero con superpoderes: también descomprime TARs y acepta URLs como origen. Usar COPY cuando no se necesiten estas funciones extra.
WORKDIR Define el directorio de trabajo para las instrucciones siguientes (RUN, CMD, ENTRYPOINT, COPY, ADD). Persiste entre capas. Diferente de cd que no persiste.
ENV Define variables de entorno disponibles durante la construcción y en el contenedor en ejecución. Ej: ENV NODE_ENV=production.
EXPOSE Documenta en qué puerto escucha el contenedor. No publica el puerto — solo es informativo. Para publicar hay que usar -p en docker run o ports: en compose.
CMD Comando por defecto al iniciar el contenedor. Puede sobreescribirse en docker run. Formato JSON recomendado: CMD ["npm", "start"].
ENTRYPOINT Define el ejecutable principal del contenedor. Más difícil de sobreescribir que CMD. Se combina con CMD: ENTRYPOINT es el comando, CMD son los argumentos por defecto.
VOLUME Declara un punto de montaje de volumen. Los datos en ese directorio persisten fuera del contenedor y se pueden compartir entre contenedores.

Ejemplo completo — app Node.js

# 1. Imagen base FROM node:9.4 # 2. Crear directorio de trabajo RUN mkdir -p /app # 3. Establecer contexto WORKDIR /app # 4. Instalar dependencias primero (capa separada → caché más eficiente) COPY package.json /app/ RUN npm install # 5. Copiar el resto del código fuente COPY . /app # 6. Exponer puerto EXPOSE 3000 # 7. Comando de arranque CMD ["npm", "start"]

Proceso de construcción — docker build

# Construir imagen con tag docker build -t my-app:1.0 . # Ver imágenes locales docker images # Ejecutar contenedor docker run -d -p 3000:3000 my-app:1.0 # Ver diferencias respecto a imagen base docker container diff <container_id>

Proceso de build: Para cada instrucción del Dockerfile, Docker crea un contenedor temporal a partir de la imagen anterior, ejecuta la instrucción, guarda el resultado como nueva capa, y elimina el contenedor temporal. Se repite hasta la última instrucción.

05 — Docker Compose

Orquestación de múltiples contenedores

Docker Compose permite definir y ejecutar aplicaciones multi-contenedor con un único fichero YAML (docker-compose.yml). Un solo comando (docker compose up) arranca todos los servicios definidos.

Términos clave de Docker-Compose — pregunta directa de examen

Término Definición
services Sección principal. Define cada uno de los contenedores/microservicios que componen la aplicación (ej: web, db, redis).
image Especifica la imagen Docker que se usará para ese servicio (de Docker Hub o registro privado). Ej: image: postgres:14.
build Ruta al Dockerfile para construir la imagen del servicio en lugar de usar una imagen ya existente. Ej: build: ./app.
ports Mapea puertos del host al contenedor. Formato: "puerto_host:puerto_contenedor". Ej: "3000:3000".
volumes Monta volúmenes o directorios del host en el contenedor para persistir datos. Ej: - pets-data:/var/lib/postgresql/data.
bind mounts Tipo de volumen que mapea un directorio específico del host directamente al contenedor. Los cambios en el host se reflejan inmediatamente (útil en desarrollo).
environment Define variables de entorno para el contenedor. Ej: POSTGRES_PASSWORD: secret.
links Conecta servicios entre sí (legacy). Permite que un servicio resuelva el nombre del otro por DNS interno. Hoy se prefiere usar redes definidas por usuario.

Ejemplo de docker-compose.yml completo

# Versión del formato de Docker Compose version: '3.5' services: web: # Servicio 1: aplicación web build: ./web # Construir desde Dockerfile en ./web ports: - "3000:3000" # host:contenedor environment: - NODE_ENV=production links: - db # puede resolver "db" por DNS db: # Servicio 2: base de datos image: postgres:14 # Usar imagen de Docker Hub volumes: - pets-data:/var/lib/postgresql/data environment: - POSTGRES_PASSWORD=secret volumes: # Declarar volúmenes usados pets-data: # Se reutiliza entre reinicios
# Levantar toda la aplicación en background docker compose up -d # Ver estado de los servicios docker compose ps # Escalar un servicio a 3 instancias docker compose up --scale web=3 # Parar y eliminar contenedores docker compose down
06 — Arquitecturas distribuidas y microservicios

Monolítico vs. Distribuido

Aplicación Monolítica

  • Un único binario en un servidor con nombre + IP fija
  • Módulos estrechamente acoplados
  • Escalar = escalar todo el sistema
  • Un fallo puede tumbar todo
  • Despliegue más simple

Aplicación Distribuida

  • Múltiples servicios débilmente acoplados
  • Servidores sin nombres amigables (UUIDs)
  • Escalar = escalar solo el servicio que lo necesita
  • Fallos aislados (circuit breaker)
  • Mayor complejidad de gestión

Patrones y atributos de una app distribuida

Componentes débilmente acoplados

Cada servicio solo conoce la interfaz pública de los otros, no su implementación interna. Durante el desarrollo se usan stubs o mocks para simular otros componentes. Clave para desarrollar en paralelo.

Stateful vs. Stateless

Stateless: no crea ni modifica datos persistentes. Fácil de escalar, mover entre nodos y reiniciar. Stateful: crea datos persistentes (bases de datos, ficheros). Más complejo de gestionar en arquitecturas distribuidas.

Descubrimiento de Servicios (DNS)

El Servicio A no sabe en qué nodo corre el Servicio B. Existe una autoridad externa (servicio DNS) que conoce la topología y devuelve la IP+puerto cuando se le pregunta. Docker y Kubernetes tienen DNS interno integrado.

Balance de Carga (Load Balancing)

Cuando un servicio tiene múltiples instancias, un load balancer distribuye las peticiones entre ellas. Algoritmo más común: round robin (asigna en orden: 1→2→3→1→2…).

Reintentos (Retry pattern)

Si una llamada falla o expira, se reintenta tras un breve espera. Cada reintento aumenta el tiempo de espera. Tras N intentos se abandona y se devuelve servicio degradado (datos en caché o vacíos).

Gestión de errores — Fail Fast

Los errores irrecuperables deben detectarse lo antes posible y hacer que el servicio falle inmediatamente. Siempre loguear en STDERR/STDOUT información útil para diagnóstico.

Redundancia

Cada componente debe ejecutarse en múltiples instancias en nodos diferentes. También la infraestructura (routers, load balancers). Objetivo: ningún punto único de fallo (SPOF).

Circuit Breaker

Evita fallos en cascada. Monitorea las llamadas al Servicio B. Si el número de fallos supera un umbral, el "interruptor se dispara" y corta las llamadas al servicio defectuoso temporalmente. Como un fusible eléctrico.

Estrategias de despliegue sin tiempo de inactividad

🔵🟢 Despliegues Azul-Verde

Azul = versión actual en producción (todo el tráfico). Se despliega Verde = nueva versión en paralelo. Se ejecutan smoke tests sobre Verde. Si pasan, el router redirige todo el tráfico a Verde. Si Verde falla, volver a Azul inmediatamente.

🐦 Canary Releases

Despliegue gradual. Versión nueva recibe inicialmente el 1% del tráfico, se monitoriza, si está bien se sube a 5%, luego más… hasta el 100%. Si algo falla en cualquier punto, se revierte. Reduce el riesgo de nuevas versiones.

Rollback

Recuperación cuando una actualización introduce errores no detectados en pruebas. Siempre debe estar planificado y probado — no es un plan B, es parte del proceso. En apps distribuidas con contenedores es sencillo: volver a la imagen anterior.

Cambios de datos irreversibles

Si hay un cambio de esquema irreversible en BD, hacerlo en 3 fases: (1) migración compatible con versión anterior, (2) despliegue del nuevo código, (3) limpieza del esquema antiguo.
