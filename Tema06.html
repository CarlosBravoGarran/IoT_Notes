01 — Por qué HTTP no es suficiente para IoT

Limitaciones de HTTP en entornos restringidos

HTTP fue diseñado para computación de propósito general en modelos cliente/servidor. Los dispositivos IoT pueden ser muy restringidos, remotos y con ancho de banda limitado. Por eso se necesitan protocolos más eficientes.

Problemas de HTTP en IoT

  • › Cabeceras HTTP son grandes (cientos de bytes)
  • › Requiere TCP — mucho overhead en handshake
  • › Modelo síncrono — cliente espera respuesta
  • › No escala bien a millones de dispositivos
  • › No apto para dispositivos dormidos/esporádicos

IoT vs M2M

Una diferencia clave: M2M puede comunicarse a través de WAN a un servidor dedicado sin ningún protocolo encapsulado. IoT se basa por definición en la comunicación entre endpoints y servicios usando Internet como tejido de red común — siempre con protocolos encapsulados.

02 — Paradigmas de mensajería

MOM (pub/sub) vs. RESTful

Publicar / Suscribir (pub/sub)

  • Comunicación mediante colas de mensajes distribuidas
  • Existe un broker central que gestiona el flujo
  • Productores y consumidores desacoplados — no se conocen entre sí
  • No necesitan conocer IP/puerto del otro extremo
  • Invariante en el tiempo — el suscriptor puede responder horas después
  • Escala muy bien a millones de dispositivos
  • Datos pueden persistir en colas (resiliencia)
  • Ejemplos: MQTT, AMQP, STOMP

RESTful HTTP

  • Servidor posee el estado de un recurso
  • Clientes acceden con GET, PUT, POST, DELETE sobre un URI
  • Sin broker — conexión directa cliente-servidor
  • Patrón síncrono de solicitud-respuesta
  • Clientes inician siempre la comunicación
  • Disfruta de la seguridad HTTPS
  • Típico de arquitecturas cliente-servidor
  • Ejemplo: APIs web, CoAP (versión ligera)

MQTT no es una cola de mensajes

Aunque usa pub/sub, MQTT no almacena mensajes por defecto. Si nadie está suscrito a un topic cuando se publica un mensaje, ese mensaje se pierde. Las colas de mensajes tradicionales (AMQP) sí almacenan. La diferencia es importante: MQTT es para tiempo real, no para almacenamiento.

03 — MQTT

MQTT — Message Queuing Telemetry Transport

"Protocolo de mensajería de publicación/suscripción, extremadamente simple y liviano, diseñado para dispositivos restringidos y redes de bajo ancho de banda, alta latencia o no confiables. Minimiza el ancho de banda de la red y los requisitos de recursos del dispositivo, garantizando confiabilidad y cierto grado de garantía de entrega."

características clave de MQTT
  • Protocolo pub/sub sobre TCP/IP
  • Orientado a conexión (cliente ↔ broker primero)
  • Cabecera fija de solo 2 bytes
  • Payload máximo: 256 MB (en práctica limitado por broker)
  • Independiente del formato de datos (JSON, binario, texto…)
  • Creado por IBM en 1993 para oleoductos remotos vía satélite
  • Estándar OASIS desde 2013, ISO/IEC 20922
  • Escala a millones de mensajes/hora y decenas de miles de editores

Flujo pub/sub en MQTT

Arquitectura MQTT — broker central
Sensor 1 (Pub) ──PUBLISH "temperatura/sala"──▶ MQTT Broker ──▶ App Dashboard (Sub)
Sensor 2 (Pub) ──PUBLISH "humedad/sala"──────▶ MQTT Broker ──▶ Base de datos (Sub)
El broker filtra mensajes por topic y los reenvía solo a los suscriptores interesados.
Editores y suscriptores no se conocen directamente — el broker es el intermediario.

Estructura del paquete MQTT

Fixed Header
2 bytes — siempre presente
Variable Header
Opcional — depende del tipo
Payload
Opcional — hasta 256 MB
El fixed header incluye: tipo de mensaje (CONNECT, PUBLISH, SUBSCRIBE…) + flags + longitud restante

Tipos de filtrado del broker MQTT

Filtrado de temas

Los clientes se suscriben a topics específicos y ramas. El broker solo reenvía los mensajes que coincidan. Ej: suscribirse a temperatura/# recibe todos los subtopics de temperatura.

Filtrado de contenido

El broker puede inspeccionar y filtrar los datos publicados. Permite gestionar datos no cifrados antes de almacenarlos o publicarlos a otros clientes.

Filtrado de tipo

El suscriptor también puede aplicar sus propios filtros al flujo de datos recibido. Analiza el contenido y decide si procesar o ignorar cada mensaje entrante.

Niveles de QoS — Calidad de Servicio

QoS 0
Fire and Forget

Entrega de mejor esfuerzo. Sin acuse de recibo. El receptor no confirma. El emisor no reintenta. Si el mensaje se pierde, se pierde.

✓ Conexiones por cable · Ancho de banda muy limitado · Datos que no son críticos
QoS 1
Al menos una vez

Garantiza entrega al menos una vez. Receptor envía PUBACK. Puede recibirse más de una vez (duplicados posibles). Más rápido que QoS-2.

✓ Uso predeterminado recomendado · Reduce costo de transmisión
QoS 2
Exactamente una vez

Garantía de entrega exactamente una vez. Protocolo de 4 pasos: PUBLISH → PUBREC → PUBREL → PUBCOMP. Más tráfico, más lento.

✓ Misión crítica · Cuando duplicados causarían fallos
04 — MQTT-SN

MQTT-SN — MQTT para redes de sensores

Derivado de MQTT diseñado para redes inalámbricas de área personal (WPAN). Misma filosofía, adaptada a: bajo ancho de banda, fallos de enlace, mensajes cortos, hardware muy restringido.

Diferencias con MQTT estándar

  • › No requiere TCP/IP — funciona sobre UDP, BLE, Zigbee, serie
  • › Topics: nombres reemplazados por IDs cortos de 2 bytes
  • › 3 mensajes CONNECT (vs 1 en MQTT) para transportar topic + mensaje
  • › Proceso de descubrimiento para encontrar gateways
  • › Soporte para clientes dormidos — mensajes en búfer hasta despertar
  • › Múltiples gateways posibles — reparto de carga

Actores de MQTT-SN

  • › Clientes — publican y se suscriben igual que en MQTT
  • › Gateway — convierte MQTT-SN ↔ MQTT. Primero se conecta al broker MQTT, luego emite ADVERTISE a los clientes
  • › Retransmisores — nodos intermedios que re-encapsulan tramas sin modificarlas
  • › Brokers — igual que en MQTT estándar

Mensaje ADVERTISE

Transmitido periódicamente por una gateway para anunciar su presencia a los clientes de la red.

Mensaje SEARCHGW

Emitido por un cliente cuando busca una gateway. Incluye parámetro de radio (número de saltos que puede seguir el mensaje).

Mensaje GWINFO

Respuesta de la gateway al SEARCHGW. Contiene el ID y dirección de la gateway. Solo se transmite cuando la solicitud viene de un cliente.

Clientes dormidos

Modo especial donde el servidor/router de borde almacena en búfer los mensajes destinados al cliente dormido y los entrega al despertar. Clave para baterías.

05 — CoAP

CoAP — Constrained Application Protocol

Protocolo del IETF (RFC 7252) creado por el grupo CoRE. Diseñado específicamente para comunicación M2M entre nodos de borde restringidos. Funciona sobre UDP (no TCP). Similar a HTTP pero hasta 10× más pequeño en cabeceras.

CoAP vs MQTT

Paradigma REST (req/resp) Pub/Sub
Transporte UDP TCP
Cabecera 4 bytes (10-20 total) 2 bytes fijos
Broker No (directo) Sí (obligatorio)
Observadores Sí (único en CoAP) Suscriptores

URI en CoAP

Comparte sintaxis con HTTP. El usuario debe conocer la dirección de antemano.

coap://host[:puerto]/[ruta][?consulta]

Rendimiento: algunas implementaciones de CoAP funcionan hasta 64 veces mejor que sus equivalentes HTTP en hardware similar.

Métodos CoAP (como HTTP)

GET

Obtener el estado actual de un recurso

PUT

Actualizar un recurso existente

POST

Crear un nuevo recurso

DELETE

Eliminar un recurso

Códigos de respuesta CoAP

2.01Created — recurso creado
2.02Deleted — recurso eliminado
2.04Changed — recurso modificado
2.05Content — respuesta con contenido
4.04Not Found — recurso no encontrado
4.05Method Not Allowed — método no permitido

Los 7 actores de CoAP

Endpoints

Orígenes y destinos de mensajes CoAP. La definición exacta depende del transporte utilizado.

Cliente

Origina la solicitud. Destino de la respuesta. Inicia la comunicación.

Servidor

Destino de la solicitud. Origen de la respuesta. Aloja los recursos.

Proxy

Endpoint al que clientes delegan solicitudes. Puede hacer cross-proxying entre CoAP y HTTP. Reduce carga de red y proporciona seguridad.

Intermediario

Cliente que actúa también como servidor hacia un servidor de origen. Un proxy es un tipo de intermediario.

Servidor de origen

El servidor donde reside físicamente el recurso solicitado. El destino final.

Observadores ⭐ — exclusivo de CoAP

Un cliente se registra para observar un recurso mediante un GET modificado. El servidor notifica automáticamente al observador cada vez que el recurso cambia de estado. Similar al modelo de suscripción MQTT. La relación termina cuando el cliente envía RST u otro GET.

Tipos de mensajes CoAP

CON (Confirmable)

Requiere ACK del receptor. Si no llega ACK, el emisor retransmite con mismo Message_ID. Mecanismo de timeout para compensar la falta de fiabilidad de UDP.

NON (Non-confirmable)

No requiere ACK. Para datos no críticos donde la velocidad es prioritaria (lecturas de sensores frecuentes).

ACK (Acknowledgement)

Confirma un CON. Puede incluir la respuesta al mensaje confirmable (respuesta piggyback) para optimizar la comunicación.

RST (Reset)

Indica que el receptor no puede procesar el mensaje. También termina una relación de observación.
