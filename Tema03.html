01 — Qué es un sistema operativo embebido

Definición y propósito

Un SO embebido es un conjunto de bibliotecas de software que cumple dos funciones principales en un sistema embebido:

1. Capa de abstracción

El software sobre el SO es menos dependiente del hardware. Facilita el desarrollo de middleware y aplicaciones sin preocuparse del hardware específico subyacente.

2. Gestión de recursos

Administra hardware y software del sistema para garantizar que todo funcione de forma eficiente y confiable. Incluye CPU, memoria, E/S y dispositivos.

⚠ El SO es opcional en sistemas embebidos — no todos los sistemas integrados tienen uno. Los más sencillos funcionan "bare metal" directamente sobre el hardware.

02 — Requisitos de un SO embebido

Qué debe cumplir un SO para IoT

🧠
Requisitos de CPU

Complejidad de operaciones muy baja. Algunas MCU funcionan a ciclos de reloj muy bajos.

💾
Requisitos de Memoria

Mínimos. Dispositivos IoT típicos tienen 5 kB a cientos de MB. Afecta RAM y almacenamiento persistente.

🔧
Características Limitadas

Funcionar sin MMU ni FPU (unidad de gestión de memoria / punto flotante). Hardware restringido.

🔌
Soporte de Plataformas

Ejecutarse en plataformas restringidas y aprovechar las capacidades de plataformas más potentes.

⚡
Eficiencia Energética

Bajo consumo. Modo de suspensión. Muchos dispositivos funcionan con baterías durante años.

🌐
Stack de Red Adaptativo

Soporte para protocolos IoT ligeros (6LoWPAN, MQTT, CoAP) adaptados a redes restringidas.

🛡️
Fiabilidad y Autonomía

Funcionar sin supervisión humana durante largos períodos. Tolerancia a fallos.

📐
API Estándar (POSIX)

Facilita el desarrollo y migración. POSIX incluye: procesos, señales, temporizadores, pipes, E/S, memoria compartida.

💻
Lenguajes Estándar

Soporte para C/C++ como mínimo. Vital para desarrollo de sistemas embebidos.

POSIX — qué incluye

POSIX.1 — Core Creación/control de procesos, señales, temporizadores, operaciones de ficheros, pipes, biblioteca C estándar, ioctl.
POSIX.1b — Tiempo Real Planificación con prioridad, señales RT, temporizadores, semáforos, paso de mensajes, memoria compartida, E/S asíncrona.
POSIX.1c — Threads Creación/control/limpieza de hilos, planificación, sincronización, manejo de señales.
03 — Tipos de SO embebidos

Monolítico · En Capas · Microkernel

Tipo 1

Monolítico

Toda la funcionalidad (kernel + drivers + middleware) integrada en un único fichero ejecutable. Módulos de código separados dentro del mismo binario en la variante modular.

  • Más difícil de reducir, modificar y depurar
  • Gran tamaño intrínseco con dependencias cruzadas
  • Variante modular: piezas de código que reflejan funciones
Linux Raspberry Pi OS
Tipo 2

En Capas (Layered)

SO dividido en capas jerárquicas (0…N). Las capas superiores dependen de la funcionalidad de las inferiores. También un único fichero grande que incluye drivers y middleware.

  • Más fácil de desarrollar y mantener que monolítico
  • Las APIs entre capas generan sobrecarga adicional
  • Afecta al tamaño y al rendimiento
TinyOS Contiki
Tipo 3

Microkernel (Cliente/Servidor)

Kernel reducido a mínima funcionalidad (solo gestión de procesos y memoria). Drivers extraídos del kernel completamente. Nanonúcleo = solo gestión de procesos.

  • Más modular y mantenible
  • Drivers corren en espacio de usuario
  • Nanonúcleo: subclase aún más reducida
FreeRTOS RIOT

04 — Principales componentes de Raspberry Pi OS

Arquitectura interna de Raspbian

Bootloader (Gestor de arranque)

Inicializa el hardware a un estado conocido y carga el kernel. 1ª etapa: programado en ROM (no modificable). 2ª y 3ª etapa: almacenadas en la tarjeta SD, ejecutadas automáticamente.

Kernel de Linux

Controladores de dispositivos, gestión de memoria, pila de red, sistema de archivos virtual, gestión de procesos, interfaz de llamadas al sistema. Al terminar de cargar ejecuta init.

Systemd (Demonios)

Se ejecuta como demonio con PID 1. Organiza tareas como unidades: servicios (.service), puntos de montaje (.mount), dispositivos (.device), sockets (.socket), temporizadores (.timer).

Shell — Bash

Interfaz de comandos. Permite supervisar y controlar el sistema. Capacidad de ejecutar scripts (colección de comandos en un fichero). Se inicia tras los demonios.

Utilidades de Shell (APT)

Comandos disponibles tras Bash: copia de ficheros, creación de directorios, APT (Advanced Packaging Tool — instalar/eliminar software). Sin ellas el sistema fallaría.

Servidor gráfico X.Org

Plataforma común para construir GUI. Maneja movimiento del ratón, teclas, dibuja aplicaciones en pantalla. Sobre él corre LXDE (Lightweight X11 Desktop Environment).

05 — Controladores de dispositivo (Device Drivers)

Funciones de los drivers

Los controladores de dispositivo son bibliotecas de software que inicializan el hardware y gestionan el acceso a él por parte de las capas superiores. Todo driver se compone de alguna combinación de estas funciones:

Hardware Startup Inicialización del hardware tras encendido o restablecimiento (PowerON/reset)
Hardware Shutdown Configurar el hardware en su estado PowerOFF
Hardware Disable Permitir que otro software desactive el hardware sobre la marcha
Hardware Enable Permitir que otro software active el hardware sobre la marcha
Hardware Acquire Permitir que otro software obtenga acceso singular (bloqueo) al hardware
Hardware Release Permitir que otro software libere (desbloquee) el hardware
Hardware Read Permitir que otro software lea datos del hardware
Hardware Write Permitir que otro software escriba datos en el hardware
Hardware Install Permitir que otro software instale nuevo hardware sobre la marcha
Hardware Unmapping Permitir eliminar bloques de datos de dispositivos de almacenamiento de hardware
06 — Gestión de procesos y tareas

Creación de tareas: fork / exec / spawn

fork

Crea una copia del espacio de memoria de la tarea principal para la tarea secundaria. La tarea secundaria hereda propiedades: código del programa y variables de la tarea principal.

exec

Elimina explícitamente del espacio de memoria de la tarea secundaria cualquier referencia al programa principal y establece el nuevo código de programa de la tarea secundaria.

spawn model

Crea un espacio de direcciones completamente nuevo para la tarea secundaria. Permite definir el nuevo programa y los argumentos directamente. Sin duplicar espacio de memoria.

Ventajas de fork/exec

Eficiencia ganada por la tarea secundaria al heredar propiedades de la tarea principal. Flexibilidad para cambiar el entorno de la tarea secundaria posteriormente.

Ventajas de spawn

No hay espacios de memoria duplicados para ser creados y destruidos. Más eficiente en memoria: sin la copia intermedia del fork/exec.

Planificador de tareas — factores de rendimiento

El planificador (scheduler) determina el orden y duración de las tareas que se ejecutarán en la CPU. En Raspberry Pi OS: FIFO simple, consciente de energía, un único planificador.

Factor Definición exacta (examen)
Response Time Tiempo para que el planificador cambie el contexto a una tarea lista. Incluye el tiempo de espera de la tarea en la cola lista.
Turnaround Time El tiempo que tarda un proceso en completarse (desde que se crea hasta que termina).
Overhead El tiempo y los datos necesarios para determinar qué tareas se ejecutarán a continuación.
Fairness ¿Cuáles son los factores determinantes en cuanto a qué procesos llegan a ejecutarse?
Throughput Procesar tantas tareas como sea posible en un período de tiempo determinado.
Starvation Donde una tarea nunca llega a ejecutarse. No ocurre cuando se intenta lograr un rendimiento máximo de la tarea.

Estados de una tarea en multitarea

LISTO

Esperando en la cola lista para ser asignada a la CPU

EN EJECUCIÓN

Usando actualmente la CPU

BLOQUEADO

Esperando un evento externo (E/S, señal, sincronización)

Señales e interrupciones

Las señales son indicadores de que un evento asíncrono ha sido generado (externo: otros procesos, hardware, temporizadores; o interno: errores de instrucción). Al recibir una señal, la tarea suspende su ejecución y cambia contexto al manejador de señales.

Sincronización (IPC)

Mecanismos para que tareas se comuniquen y sincronicen: memoria compartida (riesgo de condición de carrera), paso de mensajes, y semáforos. Las secciones críticas se protegen con Mutex (exclusión mutua) para evitar condiciones de carrera.

07 — Gestión de memoria y E/S

Responsabilidades del SO en memoria y E/S

Gestión de memoria — responsabilidades

  • › Mapeo entre memoria lógica y referencias de tareas
  • › Determinar qué procesos cargar en memoria disponible
  • › Asignación y desasignación de memoria para procesos
  • › Seguimiento del uso de memoria de componentes
  • › Garantizar coherencia de caché
  • › Garantizar protección de memoria por proceso

Gestión de E/S

  • › Capa de abstracción adicional sobre hardware y drivers
  • › Interfaz uniforme para dispositivos variados (solo a través de llamadas al sistema)
  • › Gestión de E/S síncrona y asíncrona
  • › Esquema eficiente de compartición entre múltiples procesos
  • › Mecanismo de almacenamiento en caché del búfer
08 — Middleware

Middleware en sistemas embebidos

Software de sistema que no es el kernel, los drivers ni la aplicación. Actúa como capa de abstracción que media entre aplicación y sistema subyacente. También puede mediar entre múltiples aplicaciones en red.

Adaptabilidad

Adaptar a cambios en disponibilidad de recursos

Conectividad

Comunicación transparente con otras aplicaciones

Flexibilidad

Configurable según requisitos y limitaciones

Portabilidad

Ejecutarse en diferentes sistemas embebidos

Seguridad

Acceso autorizado a recursos del sistema

tipos de middleware en sistemas embebidos
  • Core middleware (base del sistema)
  • Middleware complejo específico del mercado
  • Mensajería y comunicación complejas
  • Mensajería distribuida orientada a mensajes (MOM)
  • Transacción distribuida
  • Middleware de controlador de red
  • ORB (Object Request Broker)
  • RPC (Remote Procedure Calls)
