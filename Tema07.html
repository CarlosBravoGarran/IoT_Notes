01 — Qué es un microservicio

Definición y principio fundamental

"Un microservicio proporciona una capacidad empresarial o de plataforma a través de una API, un contrato de datos y una configuración bien definidos. Proporciona esta función y solo esta función. Hace una cosa y la hace bien."

El prefijo "micro" no se refiere al tamaño del servicio, sino al alcance de su funcionalidad. Un microservicio puede ser grande internamente, pero su responsabilidad hacia el exterior está perfectamente delimitada. Esta es la base de toda la arquitectura.

El problema con los monolitos

  • Todo el código compilado en un único binario grande
  • Un bug exige reconstruir, re-probar y re-desplegar todo
  • Con el tiempo, imposible de mantener y evolucionar
  • Escalar = escalar todo el sistema completo
  • Módulos fuertemente acoplados y dependientes
  • Un equipo toca código del otro constantemente

La solución con microservicios

  • Cada servicio es un proceso independiente
  • Divididos verticalmente según capacidad de negocio
  • Comunicación por red (mensajes ligeros)
  • Desplegables de forma independiente
  • Cada equipo posee un servicio de principio a fin
  • Escalar solo lo que necesita escalado
02 — Características de los microservicios

Las 9 características fundamentales

01
Autónomo

Capaz de existir independientemente. Responde, reacciona y se desarrolla de forma independiente del todo. Unidad autónoma de funcionalidad.

02
Aislado

Separado de los demás. Puede ejecutarse en diferentes lugares y en diferentes momentos. Dependencias poco acopladas con otros servicios.

03
Elástico

Capaz de volver a su forma original después de expandirse o comprimirse. Escala adecuadamente según el escenario de uso. Escala dinámica.

04
Resiliente

Capaz de mantenerse en funcionamiento después de un error. Tolerante a fallos. Proporciona un plazo razonable de recuperación si algo sale mal.

05
Responsivo

Rápido para responder o reaccionar. Brinda un desempeño razonable dado el escenario de ejecución. No bloquea a sus consumidores.

06
Orientado a Mensaje

Software que conecta sistemas separados llevando y distribuyendo mensajes entre ellos. API + contrato de datos bien definidos para la interacción.

07
Programable

Un plan de tareas que se realizan para lograr un resultado específico. Invocable mediante API y mensajes de forma predecible y repetible.

08
Configurable

Diseñado para adaptarse a diferentes configuraciones o propósitos. Proporciona interfaz/consola administrativa. Reutilizable en distintos sistemas.

09
Automatizado

Funciona sin control directo de una persona. Cultura DevOps: equipos multifuncionales (dev + ops) responsables del ciclo de vida completo.

03 — Elementos internos de un microservicio

Capas de un microservicio

Un microservicio bien construido sigue la misma separación de capas que las aplicaciones monolíticas, pero a escala reducida. La industria se ha decantado por REST sobre HTTP con JSON como formato estándar.

Capa / Elemento Función Tecnología habitual
Capa de Protocolo Define el protocolo de red y mecanismo de comunicación del microservicio. Define los endpoints visibles en red. HTTP + REST, TCP, SOAP/WSDL
Modelos (Data Contract) Definen la estructura de los datos que entran y salen. Se serializan/deserializan en la capa de protocolo y acceso a datos. JSON (reemplazó a XML por más compacto)
Capa de Servicio Donde ocurre el trabajo real: reglas de negocio, flujos de trabajo, cálculos. Implementa la funcionalidad del microservicio. Lógica de dominio, clases de servicio
Capa de Acceso a Datos Encapsula el acceso a almacenamiento: caché, relacional, documentos, blob, colas. Cada microservicio con su propio almacén. ORM, drivers BD, Redis, S3
Capa SDK (opcional) SDK específicos del lenguaje para facilitar la adopción de la API por parte de otros equipos y servicios. Python SDK, Java SDK, npm package
API Gateway Proxy entre clientes y microservicios. Proxy inverso, enrutamiento, terminación SSL, auth, rate limiting, logging centralizado. Kong, AWS API Gateway, nginx

Parámetros en URLs REST

Ruta de recursos GET /profiles/users/id/99999 → devuelve el perfil del usuario 99999
Parámetros de consulta GET /profiles/users?location=MA → devuelve array JSON de usuarios en MA
04 — Tipos de microservicios IoT

Las 5 capacidades de una plataforma IoT

Para implementar plenamente una solución IoT, se necesitan varias capacidades implementadas mediante microservicios especializados:

Tipo 01

Gestión de dispositivos

El dispositivo, tras inicializarse, establece relación con el entorno cloud mediante su identificador único (número de serie, UUID). La empresa recibe notificación de que el dispositivo está activo. Permite enviar comandos: actualizaciones de software, actualización de cachés de datos locales.

Tipo 02

Ingestión de telemetría

Los dispositivos pueden enviar varios mensajes por segundo. Con cientos o miles de dispositivos → decenas de miles o millones de mensajes al día. La plataforma cloud proporciona ingestión de gran volumen con un único punto final lógico.

Tipo 03

Transformación y almacenamiento

Una vez que llegan los mensajes, la nube selecciona, transforma y enruta los datos a varios medios de almacenamiento para archivarlos y prepararlos para procesamiento posterior (series de tiempo, SQL, NoSQL, blob).

Tipo 04

Estado y notificaciones

Visualización del estado del conjunto de mensajes en tiempo real a través de componentes de UI tabulares o gráficos. Notificaciones en tiempo real cuando mensajes contienen información de estado de alerta.

Tipo 05

Analítica y visualización de datos

El valor de recopilar datos continuamente es construir un registro histórico para análisis y obtener información empresarial. Técnicas: almacén de datos tradicional, map-reduce, análisis predictivo. La escala en IoT se mide en petabytes, exabytes y zettabytes.

05 — Patrones de implementación

Cómo implementar microservicios IoT

Patrón 1

Base de datos por microservicio

Cada microservicio tiene su propio almacén de datos para evitar acoplamiento fuerte en la capa de datos. Pueden compartir la misma BD física pero usar esquemas/colecciones/tablas separados.

Ventajas
  • Sin acoplamiento en capa de datos
  • Cada servicio elige su BD óptima
Inconvenientes
  • Transacciones distribuidas complejas
  • Consistencia eventual
Patrón 2

Event Sourcing (Abastecimiento de eventos)

En lugar de guardar el estado actual de una entidad, se guarda cada evento de cambio de estado como serie inmutable. El estado actual se deduce reprocesando todos los eventos.

Ventajas
  • Atomicidad en sistemas escalables
  • Historial completo + viaje en el tiempo
  • Microservicios débilmente acoplados
Inconvenientes
  • Lectura compleja → necesita patrón CQRS
  • Mayor complejidad general
  • Gestionar eventos duplicados/perdidos
Patrón 3

Backends para Frontends (BFF)

Cada interfaz de usuario (web, móvil, Smart TV…) obtiene su propio backend personalizado. Actúa como fachada para los microservicios downstream, reduciendo llamadas "chatty" entre UI y servicios.

Ventajas
  • Optimizable por tipo de UI
  • Mayor seguridad
  • Menos llamadas UI↔microservicio
Inconvenientes
  • Duplicación de código entre BFFs
  • Proliferación si hay muchas UIs
  • No debe contener lógica de negocio
Patrón 4 — API Gateway

API Gateway

Se sitúa entre clientes y microservicios. Proxy inverso que enruta solicitudes al microservicio correcto. También agrega respuestas de múltiples servicios. Gestiona preocupaciones transversales centralizadas.

Ventajas
  • SSL, auth, rate limiting centralizados
  • Reduce viajes de ida y vuelta cliente
  • Desacoplamiento front-back
Inconvenientes
  • Punto único de fallo potencial
  • Mayor latencia por salto extra
  • Cuello de botella si no escala

Consumer-Driven Contract Testing

En arquitecturas con muchos microservicios desarrollados por equipos distintos, las pruebas de integración son un reto. La solución: el equipo consumidor escribe contratos explícitos (solicitud + respuesta esperada). El microservicio proveedor incluye esos contratos en su suite de pruebas automatizadas. Si el proveedor cambia su API incompatiblemente, los contratos lo detectan automáticamente.

06 — Ventajas e inconvenientes

Por qué usar microservicios

⚡
Mayor velocidad

Desarrollo y despliegue más rápido por equipo independiente

📐
Mejor escalado

Escalar solo el servicio que lo necesita (granular y horizontal)

🔄
Modernización iterativa

Migrar módulo a módulo sin reescribir todo el sistema

☁️
Ecosistema moderno

Aprovecha cloud, contenedores, DevOps y serverless

🧠
Menos carga cognitiva

Cada dev solo debe entender su pequeño servicio

🛡️
Fallos aislados

Un servicio caído no tumba toda la aplicación

problemas a considerar al implementar microservicios
  • Mayor número de partes móviles (servicios, BDs, procesos, contenedores)
  • La complejidad pasa del código a la infraestructura
  • Proliferación de mensajes y tráfico de red entre servicios
  • Gestionar la seguridad del sistema completo es un desafío
  • Diseñar el sistema completo es más difícil que un monolito
  • Introduce las complejidades de los sistemas distribuidos
  • Requiere cultura DevOps madura antes de emprender la migración
  • Pruebas de extremo a extremo más frágiles y costosas
07 — Modelos de despliegue de negocio en IoT

On-premises · Managed · Hosted · IaaS · PaaS · SaaS

En IoT hay que decidir qué capas gestiona el cliente (Consumer) y cuáles gestiona el proveedor (Provider). Las capas son siempre las mismas: Aplicaciones+Datos, Runtime, Middleware, SO, Virtualización, Servidor+Red, Refrigeración.

Modelo Quién gestiona qué Descripción Ejemplo IoT
On-premises Consumer gestiona TODO Toda la infraestructura física y lógica está en las instalaciones del cliente. Máximo control, máxima responsabilidad. Fábrica con servidores propios para IIoT
Managed Consumer: Apps+Datos+Runtime+MW+SO
Provider: Virtualización+Servidor+Red		 El proveedor gestiona el hardware físico. El cliente gestiona todo el software. Servidor dedicado alquilado
Hosted Provider: Virtualización+Servidor+Red+SO
Consumer: Apps+Datos+Runtime+MW		 Proveedor gestiona además el SO base. El cliente gestiona middleware y aplicaciones. VPS en OVH, Hetzner
IaaS Consumer: Apps+Datos+Runtime+MW+SO
Provider: Virtualización+Servidor+Red		 Infraestructura como Servicio. El cliente gestiona desde el SO hacia arriba. Proveedor gestiona el hardware virtualizado. AWS EC2, Azure VM, Google Compute Engine
PaaS Consumer: Apps+Datos
Provider: Todo lo demás (Runtime, MW, SO, Virt., Servidor)		 Plataforma como Servicio. El cliente solo gestiona su aplicación y sus datos. El proveedor gestiona toda la plataforma. AWS IoT Core, Azure IoT Hub, Google Cloud IoT
SaaS Consumer: solo usa la App
Provider: gestiona TODO		 Software como Servicio. El cliente solo usa la aplicación. El proveedor gestiona absolutamente todo. ThingSpeak, Ubidots, Grafana Cloud

08 — Principios de diseño

Separación de Intereses (SoC) y DevOps

Separación de Intereses (SoC)

Principio de diseño que separa la implementación en capas distintas, cada una abordando una preocupación separada. Se logra mediante encapsulación de información e interfaces bien definidas.

En microservicios: las capas de negocio y datos se cortan verticalmente a lo largo de costuras funcionales. Cada servicio tiene su propio modelo de dominio y API.

Arquitectura típica: 3 capas — Presentación (UX) · Aplicación (negocio) · Datos (persistencia).

Cultura DevOps

Promueve la colaboración e integración de equipos de desarrollo y operaciones. El equipo que diseña un microservicio también lo despliega y mantiene.

Equipos multifuncionales: arquitectos + desarrolladores + testers + ingenieros de operaciones. Todos responsables del ciclo de vida completo del producto.

Sin una canalización de CI/CD completamente automatizada, la proliferación de microservicios se vuelve inmanejable. DevOps no es opcional — es un prerrequisito.
