01 — Arquitectura de una solución IoT

Los 4 bloques fundamentales

Una solución IoT completa siempre se organiza en cuatro capas funcionales. Memorizar qué elementos pertenecen a cada capa es esencial para el examen.

01 / Dispositivo

Dispositivos ("Things")

Sensores y actuadores físicos. Nodos que interactúan con el mundo físico.

  • Embedded Device / System
  • Sensor
  • Actuador / Relay / Servo Motor
  • Termostato, cámara, iBeacon
  • Intelligent mousetraps
  • GSMchips
02 / Comunicación

Protocolos de comunicación

Cómo los datos viajan desde el dispositivo hacia la plataforma.

  • Zigbee / Bluetooth / Wi-Fi
  • LoRaWAN / GSM
  • MQTT / CoAP / AMQP
  • REST API
  • Message Router
03 / Pasarela

Gateway (Puertas de enlace)

Traducen redes no-IP a protocolos IP. Doble conectividad.

  • Message Broker (K)
  • Device Manager
  • GSM + Bluetooth
  • Wi-Fi + XBee
  • LoRaWAN + Ethernet
04 / Plataforma

Plataforma IoT

Orquestador en la nube. Almacena, procesa y expone datos.

  • Storage and Data Management
  • Rule Engine
  • REST API / Microservices
  • Visualización
  • Device Manager (L)
02 — Paradigmas de computación distribuida

Edge · Fog · Cloud

Tres paradigmas complementarios que forman una jerarquía. El examen pregunta sus diferencias y casos de uso.

☁ Cloud Computing
Centros de datos centralizados en cualquier parte del mundo. Almacenamiento masivo, análisis de big data, aplicaciones empresariales generales. Mayor latencia.
Lejano · Centralizado
🌫 Fog Computing
Capa intermedia entre Edge y Cloud. Nodos de niebla (dispositivos, servidores, gateways) ubicados en la red local. Automatización industrial, smart cities. Computa dentro de la red de área local.
Intermedio · Red local
⚡ Edge Computing
Procesamiento directamente en el dispositivo IoT o muy cerca de él. Decisiones en tiempo real, baja latencia. Datos computados en el sensor, sin trasladarse. Conserva toda la información localmente.
Cercano · Tiempo real

Diferencia clave: Fog vs Edge

FOG
  • Desplaza operaciones a nodos de niebla en la red local
  • Computa dentro de la red de área local
  • Puede gestionar más información en intervalos de tiempo
EDGE
  • Ocurre directamente en el dispositivo IoT
  • Sin trasladarse a centros de datos o nube
  • Conserva datos y cálculo donde se generó
03 — Componentes de una plataforma IoT

Bloques funcionales de la plataforma

La plataforma IoT (instalada en la nube / VPS Linux) organiza el procesamiento en módulos especializados interconectados por un bus de mensajes.

Interfaz periférica / Message Broker

Se comunica con dispositivos heterogéneos (Wi-Fi, Bluetooth, LoRaWAN). Consolida datos en el bus de mensajes. Actúa como coordinador central.

Message Router / Comunicaciones

Enriquece mensajes, convierte formatos (CSV→JSON, binario→texto), deduplica paquetes y controla el flujo en el bus.

Storage — Series temporales

Almacena datos secuencialmente. Para almacenamiento provisional; no pensado para conjuntos de datos muy grandes.

Rule Engine

Supervisa el bus y ejecuta acciones basadas en reglas. Ejemplo: "enviar alerta cuando el paquete contenga X". Activado por eventos.

REST API

Para funciones que no necesitan tiempo real. Sensores con lectura cada 15 min usan REST. Puede activar reglas y flujos de trabajo.

Microservices

Notificaciones, autenticación, pagos, captcha, etc. Agrupados y empaquetados para reutilización dentro y fuera de la plataforma.

Device Manager

Gestiona 50+ dispositivos: estado activo/inactivo, nivel batería, actualizaciones OTA, derechos de acceso, sesiones.

App & User Manager

Contraseñas, credenciales, API keys, derechos de acceso para aplicaciones y usuarios upstream.

Fog / Edge nodes

Nodos físicos/virtuales de datos, nodos de niebla (dispositivos, servidores, gateways), servicios de supervisión, procesadores y gestor de recursos.

04 — Protocolos IoT por capa OSI

Stack de protocolos

Los protocolos IoT permiten la comunicación entre dispositivos y servidores. Organizados por capa de red:

Protocolo Capa Descripción Uso típico en IoT
802.11 Wi-Fi Enlace LAN inalámbrica. Bandas 2.4/5/60 GHz. Dispositivos domésticos, gateways
Zigbee / 802.15.4 Enlace LR-WPAN. Bajo costo y baja velocidad. Base de Zigbee, 6LoWPAN. Sensores de baja potencia
LoRaWAN Enlace Long Range. Gran alcance, bajo consumo (LPWAN). Sensores remotos, agricultura
Bluetooth / BLE Enlace Corto alcance. BLE = muy bajo consumo. Wearables, iBeacon, teléfonos como gateway
GSM / 2G-4G Enlace Redes celulares. Amplia cobertura. Dispositivos remotos con SIM
IPv4 / IPv6 Red IPv4: 32 bits. IPv6: 128 bits (3.4×10³⁸ dir.). Identificación y enrutamiento
6LoWPAN Red IPv6 sobre redes de baja potencia. Dispositivos restringidos. Sensores con IPv6
TCP / UDP Transporte TCP = fiable, orientado conexión. UDP = sin conexión, más ligero. TCP para MQTT; UDP para CoAP
MQTT Aplicación Pub/sub extremadamente ligero. Sobre TCP. QoS 0/1/2. Sensores con conexión limitada
CoAP Aplicación REST sobre UDP para dispositivos restringidos. Similar a HTTP pero 10x más pequeño. M2M, nodos de baja potencia
AMQP Aplicación Cola de mensajes. Componentes rígidos para mensajería empresarial. Plataformas empresariales
HTTP / REST Aplicación Cliente-servidor. Sin estado. Base de Internet. APIs de plataforma, polling periódico

06 — Anatomía de un dispositivo IoT

Componentes clave de un dispositivo

Sensores y Actuadores

Sensores captan datos del entorno (temperatura, movimiento, luz) → señales eléctricas. Actuadores transforman señales eléctricas → acciones físicas.

CPU

Cerebro del dispositivo. Procesa datos de sensores, toma decisiones, gestiona memoria y comunicación.

Memoria

RAM + almacenamiento persistente. En dispositivos IoT típicos: 5 kB a cientos de MB. Extremadamente limitada.

Interfaz de comunicación

Permite conexión a internet u otros dispositivos. Wi-Fi, Bluetooth, celular, etc. A veces un gateway hace este trabajo.

Unidad de alimentación

Suministra energía. Crítica en IoT: muchos dispositivos funcionan con baterías → bajo consumo es prioridad de diseño.

Gateway (cuando es necesario)

No todos los dispositivos tienen conectividad IP directa. El gateway conecta redes no-IP (Bluetooth, Zigbee) con redes WAN/Internet.
